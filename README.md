# VisionGuard V1

VisionGuard V1 is a project plan and starter architecture for an AI-powered defect detection web app that runs on **GitHub Pages**, uses **Firebase** for image/data storage, and uses **Google Colab** for model training.

## Goal
Build a two-page web application for industrial defect detection:

1. **Live Detection page**
   - Runs on a phone browser.
   - Opens the rear camera.
   - Uses an in-browser ML model to detect defects.
   - Draws a bounding box around a defect.
   - Starts with one target defect: **missing breather**.

2. **Dataset & Training page**
   - Lets users upload and label defect images.
   - Stores images and metadata in Firebase.
   - Exports dataset metadata for training.
   - Feeds Google Colab training workflows.

## Recommended Stack

### Frontend
- **HTML, CSS, JavaScript**
- **Tailwind CSS** for fast UI styling
- **Vite** for local development and static builds

### Browser AI
- **TensorFlow.js** for in-browser inference
- Start with a lightweight detector/classifier pipeline:
  - Phase 1: classification + guided capture
  - Phase 2: object detection with bounding boxes
- Export trained model as:
  - `model.json`

- **Firebase Firestore** for labels, sessions, predictions, and dataset metadata

### Training Environment
- **Google Colab** for dataset preparation and model training
- Python libraries:
  - TensorFlow / Keras
  - OpenCV
  - Pandas
  - Firebase Admin SDK
  - Albumentations (optional)

## Product Scope

### Page 1: Live Defect Detection
**Purpose:** Inspect a component using the phone camera in real time.

#### Key features
- Camera access with `navigator.mediaDevices.getUserMedia()`
- Rear-camera preference on mobile devices
- Video stream with canvas overlay
- On-device model inference in a loop
- Bounding box rendering for detected defects
- Confidence threshold control
- Snapshot capture when a defect is detected
- Upload event logs to Firebase

#### Detection flow
1. User opens the detection page.
2. Browser requests camera permission.
3. Live feed starts.
4. Model processes frames at a controlled interval.
5. If `missing_breather` is detected above threshold:
   - draw red box
   - show confidence label
   - optionally vibrate phone / play sound
   - save snapshot + metadata to Firebase

#### Metadata to save per detection
- `defectType`
- `confidence`
- `timestamp`
- `deviceId` or session id
- `imageUrl`
- `boundingBox`
- `inspectionMode`
- optional `gpsLat`, `gpsLng` if permission granted

### Page 2: Defect Dataset Builder
**Purpose:** Create and manage the image dataset used to improve the model.

#### Key features
- Upload one or many images
- Label each image as:
  - `normal`
  - `missing_breather`
  - future defect classes
- Add notes and part metadata
- Browse existing dataset images
- Filter by label, date, or status
- Mark images for training/validation/test split
- Export dataset manifest for Colab

#### Dataset curation workflow
1. User uploads images from phone or desktop.
2. Image is stored in Firebase Storage.
3. Firestore record is created with label and metadata.
4. Dataset browser shows the image in a review grid.
5. Reviewer can relabel, approve, or archive bad samples.
6. Manifest is generated for training in Colab.

## Firebase Data Model

### Firestore collections

#### `defect_images`
Stores the master record for uploaded images.

Example fields:
- `label`: `missing_breather`, `normal`
- `storagePath`
- `downloadUrl`
- `source`: `manual_upload` or `live_capture`
- `status`: `new`, `approved`, `rejected`
- `split`: `train`, `val`, `test`
- `createdAt`
- `updatedAt`
- `notes`
- `bbox`: `{ x, y, width, height }` if annotated

#### `detections`
Stores events generated by the live detector.

Example fields:
- `defectType`
- `confidence`
- `capturedAt`
- `imageUrl`
- `bbox`
- `sessionId`
- `deviceInfo`
- `location`

#### `training_runs`
Stores training metadata and exported model information.

Example fields:
- `runName`
- `labels`
- `datasetVersion`
- `startedAt`
- `finishedAt`
- `metrics`
- `modelVersion`
- `modelPath`

## Model Strategy

### Phase 1: Fastest working MVP
Use a lightweight **image classifier** to decide whether the current view contains a missing breather.

Why start here:
- easier to train with fewer examples
- simpler annotation requirements
- faster browser inference on phones

Limitation:
- classification alone does not produce a precise bounding box

### Phase 2: True defect localization
Upgrade to an **object detection** model so the app can draw a box around the defect.

Candidate options:
- TensorFlow object detection export to TFJS
- YOLO model exported to ONNX or TFJS
- Custom lightweight SSD/MobileNet detector

Recommendation:
- Start with **classification MVP**
- Collect more annotated images
- Move to **object detection** once enough bounding-box labeled data exists

## Google Colab Workflow

### Notebook responsibilities
1. Authenticate to Firebase using a service account
2. Download approved dataset records
3. Build train/validation/test splits
4. Apply augmentation
5. Train the model
6. Evaluate metrics
7. Export model for web inference
8. Save model artifacts for GitHub Pages deployment

### Colab pipeline outline
- Load Firestore metadata
- Download image files from Firebase Storage
- Normalize labels
- Resize images
- Train transfer-learning model
- Save best checkpoint
- Convert to TensorFlow.js format
- Produce a deployment bundle and metrics summary

### Suggested metrics
- precision
- recall
- F1 score
- confusion matrix
- inference latency on mobile

## MVP Build Order
For the first usable version, build in this order:

1. Firebase project setup
2. Static two-page frontend scaffold
3. Dataset upload and label flow
4. Colab notebook for classifier training
5. Browser model loading and live prediction
6. Detection logging and image capture
7. Detector upgrade for bounding boxes

## Risks and Mitigations

### 1. GitHub Pages cannot run server-side training
**Mitigation:** keep training in Google Colab and only deploy static artifacts to GitHub Pages.

### 2. Real-time detection may be slow on older phones
**Mitigation:** use smaller input sizes, quantized models, and reduced frame-rate inference.

### 3. Bounding boxes require annotated data
**Mitigation:** start with classification, then add box annotations after enough examples are collected.

### 4. Firebase costs can grow with image uploads
**Mitigation:** compress images before upload, archive old images, and use lifecycle rules.

## Definition of Done for MVP
The MVP is complete when:
- A phone can open the GitHub Pages site.
- The live page can access the camera.
- The app can run a browser ML model.
- The app can detect `missing_breather` at a basic level.
- The dataset page can upload labeled images to Firebase.
- Google Colab can train a model from Firebase data.
- The exported web model can be loaded back into the site.

This repository now includes a static starter prototype for GitHub Pages:

- `styles.css` for the shared UI system
- `js/detect.js` for camera access and a mock detection loop

### Run locally
Because the app uses ES modules and camera APIs, serve it with a simple local web server instead of opening the files directly.

Example options:
- `python -m http.server 4173`
- `npx serve .`

Then open:
- `http://localhost:4173/index.html`
- `http://localhost:4173/data.html`

### Current prototype limitations
- The detector is currently mocked and does not yet load a real TensorFlow.js model.
- Dataset entries are stored in browser local storage until Firebase is wired in.
- Exported manifests use base64 image previews for quick prototyping.
